# 架构与设计取舍

## 为什么选择 DuckDB

RepoPulse 面向个人分析、作品集和小型开源团队。嵌入式 DuckDB 不需要单独部署数据库服务，能够直接执行窗口函数、分位数和 Cohort SQL，也便于将完整可复现数据快照作为单个文件交付。

当采集规模扩大到数百个大型仓库时，可以保留指标接口，将明细存储替换为云数仓或 Spark/Parquet 管道。

## 增量策略

Issue 和 Commit 接口使用数据库中的最大更新时间作为下一次游标；Pull Request 按更新时间倒序读取，遇到早于游标的数据后停止。所有游标回退 5 分钟并依赖主键 upsert 去重。

Release 数量通常较少，当前版本每次全量读取并幂等覆盖。

## 分层边界

```text
github_client.py  只负责 HTTP、分页、限流与 API 语义
storage.py        只负责表结构、主键和持久化
pipeline.py       编排增量采集和运行审计
metrics.py        统一维护可解释的 SQL 指标口径
app.py            展示指标，不重复实现业务计算
```

这个边界允许数据源、存储或界面分别替换，并让指标能够被命令行、测试和网页共同复用。

## 数据可信度与迁移

GitHub 客户端为每类分页接口记录本次页数、原始记录数和是否达到分页上限。
采集管道把这些证据写入 `collection_coverage`，指标层只读取已经持久化的覆盖结论，
因此命令行、报告和页面看到的是同一份数据可信度状态。

`history_complete` 采用保守语义：增量实体一旦发生历史截断，就不会因为后续一次较小的
成功刷新而自动恢复。Release 属于全量替换实体，可以在完整刷新后恢复。

数据库使用 `schema_migrations` 记录已应用的编号迁移。新版本启动时会在事务中依次执行
缺失迁移，使旧 DuckDB 快照可以升级，同时避免重复执行已完成的结构变更。

## 隐私与安全

- 仅采集公开仓库中本来就公开的活动数据。
- GitHub Token 只通过请求头传递，不写入表或日志。
- 示例数据完全由固定随机种子生成，不对应真实个人。
